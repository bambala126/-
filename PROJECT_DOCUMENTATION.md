# Проект «Древовидный блокнот»

## 1. Назначение проекта

«Древовидный блокнот» — это настольное приложение на C++/Qt, предназначенное для хранения и редактирования иерархической базы данных узлов.

Каждый узел может:

- иметь родительский узел;
- быть обычным текстовым узлом или каталогом;
- содержать имя;
- содержать набор атрибутов;
- содержать текстовое наполнение;
- принадлежать владельцу;
- иметь права доступа для пользователей и групп.

Идея проекта — совместить в одной программе:

- дерево разделов;
- хранение текстов;
- метаданные узлов;
- разграничение доступа;
- администрирование пользователей и групп.

Проект реализован на Qt Widgets и использует базу данных SQLite.

## 2. Основная идея работы программы

Пользователь входит в систему через окно авторизации.

После успешного входа открывается главное окно:

- слева находится дерево узлов;
- справа открываются вкладки с содержимым выбранных узлов;
- для каждого текстового узла можно редактировать:
  - текст;
  - атрибуты;
- каталог может содержать дочерние узлы и каталоги;
- для узлов и каталогов действуют права доступа.

Если пользователь является администратором, ему дополнительно доступен интерфейс:

- управления пользователями;
- управления группами;
- управления правами доступа.

## 6. Слой данных: `DatabaseManager`

Файл: [databasemanager.cpp](/C:/Users/admin/Documents/TreeNote/src/databasemanager.cpp)

`DatabaseManager` — это центральный класс проекта, работающий по паттерну singleton.

### Почему выбран singleton

В приложении нужна одна общая точка доступа к:

- соединению с базой;
- текущему пользователю;
- проверке прав;
- SQL-операциям.

Это упрощает архитектуру и убирает необходимость передавать соединение в каждый класс.

### 6.1. Какие структуры данных используются

В [databasemanager.h](/C:/Users/admin/Documents/TreeNote/include/databasemanager.h) описаны:

- `NodeData` — данные узла
- `UserData` — данные пользователя
- `GroupData` — данные группы
- `AccessRightData` — запись о праве доступа

### 6.2. Открытие базы

Метод: `openDatabase(const QString& path)`

Что делает:

- подключает драйвер `QSQLITE`;
- открывает файл базы;
- включает `foreign_keys`;
- включает режим `WAL`;
- создаёт схему таблиц, если базы ещё нет;
- создаёт пользователя `admin/admin`, если в базе нет пользователей.

### Почему включён `WAL`

`WAL` (`Write-Ahead Logging`) улучшает устойчивость SQLite при чтении и записи и делает работу с базой более удобной при повторных операциях.

### Как работает окно входа

Пользователь вводит:

- логин;
- пароль.

При нажатии кнопки входа вызывается `DatabaseManager::authenticateUser(...)`.

### Как работает проверка пароля

Пароль:

1. берётся из поля ввода;
2. хешируется методом `hashPassword(...)`;
3. сравнивается с `password_hash` в базе.

### Почему хранится хеш, а не пароль

Это базовое требование безопасности:

- база не хранит пароль в открытом виде;
- в таблице хранится только его вычисленное значение.

### Какой алгоритм используется

Используется `SHA-256` с фиксированным префиксом-солью.

Это не идеальная промышленная схема, но для учебного проекта это существенно лучше, чем хранение паролей как обычного текста.

## 10. Модель дерева: `TreeModel`

Файлы:

- [treemodel.h](/C:/Users/admin/Documents/TreeNote/include/treemodel.h)
- [treemodel.cpp](/C:/Users/admin/Documents/TreeNote/src/treemodel.cpp)

`TreeModel` — это мост между таблицей `nodes` и `QTreeView`.

### Зачем нужен `QAbstractItemModel`

Qt не умеет сам превращать произвольную таблицу SQL в удобное древовидное представление с логикой доступа и drag-and-drop.

Поэтому реализована собственная модель.

### 10.1. `TreeItem`

`TreeItem` — это внутренний объект, представляющий один узел в памяти.

Он хранит:

- `NodeData`
- указатель на родителя
- список дочерних элементов

### 10.2. `TreeModel::loadChildren`

Рекурсивно загружает дерево из базы.

Алгоритм:

1. запрашивает детей заданного `parentId`;
2. создаёт `TreeItem` для каждого узла;
3. если узел — каталог, загружает его потомков.

### Почему это удобно

Так дерево полностью строится в памяти и быстро отображается в интерфейсе.

### 10.3. Drag-and-drop

Модель поддерживает перенос узлов мышью.

При переносе:

- формируется MIME-данные с `nodeId`;
- вызывается `DatabaseManager::moveNode(...)`;
- после успешного переноса дерево перезагружается.

### Защита при переносе

Перенос не допускается, если:

- пользователь не имеет `can_write` на узел;
- пользователь не имеет `can_create_children` на новый родительский каталог;
- узел пытаются перенести в себя;
- узел пытаются перенести в собственное поддерево.

Это защищает древовидную структуру от циклов и нарушений прав.

## 16. Логика проверки прав

Метод: `hasRight(int nodeId, int userId, bool isAdmin, const QString& right)`

### Алгоритм

1. Если пользователь — администратор, доступ разрешается сразу.
2. Если пользователь — владелец узла, доступ разрешается.
3. Если у узла нет явных правил, право наследуется от родителя.
4. Если это корневой узел без правил, по умолчанию разрешены:
   - `can_see`
   - `can_read`
5. Проверяются пользовательские правила.
6. Проверяются групповые правила.
7. Если явные правила есть, но ни одно не подходит — доступ запрещается.

### Почему наследование важно

Без наследования пришлось бы вручную задавать права каждому узлу.

Наследование позволяет каталогу задавать общую политику для вложенных элементов.

## 17. Защита реализована не только в UI, но и в слое БД

Это принципиально важный момент.

Даже если кнопка в интерфейсе по ошибке окажется доступна, слой БД всё равно перепроверит права.

Проверки встроены в:

- создание узла;
- изменение узла;
- удаление узла;
- перенос узла;
- сохранение содержимого;
- чтение содержимого;
- просмотр прав;
- изменение прав;
- удаление прав.

### Почему это правильно

Без этого защита была бы “косметической” и зависела бы только от GUI.

Сейчас ограничения реализованы на двух уровнях:

- интерфейс;
- бизнес-логика.

## 18. Рекурсивное применение прав к поддереву

В проект добавлена отдельная логика применения права ко всему поддереву.

### Как это работает

1. Пользователь открывает окно прав доступа.
2. Указывает:
   - пользователя или группу;
   - флаги прав.
3. Отмечает чекбокс применения ко всему поддереву.
4. Программа получает список всех узлов ниже выбранного каталога.
5. Для каждого узла:
   - ищет существующее правило;
   - если правило есть — обновляет;
   - если правила нет — создаёт.

### Как получаются все потомки

С помощью рекурсивного SQL-запроса `WITH RECURSIVE`.

### Почему это важно

Так пункт ТЗ про каталог, который “может устанавливать права доступа для вложенных узлов и каталогов”, реализован явно, а не только через наследование.

## 21. Поиск

Файлы:

- [searchdialog.cpp](/C:/Users/admin/Documents/TreeNote/src/searchdialog.cpp)
- [searchdialog.ui](/C:/Users/admin/Documents/TreeNote/ui/searchdialog.ui)

### Какие виды поиска есть

- по имени узла;
- по атрибутам;
- по содержимому.

### Как работает поиск

Пользователь вводит строку и выбирает, где искать.

Программа выполняет SQL-запросы:

- `name LIKE ?`
- `attributes LIKE ?`
- `content LIKE ?`

Затем:

- объединяет результаты;
- удаляет дубли;
- отображает список найденных узлов.

### Почему поиск сделан через `LIKE`

Это самый простой и понятный способ для учебного проекта:

- легко реализуется;
- не требует отдельного полнотекстового индекса;
- хорошо подходит для небольших и средних данных.

## 22. Быстрый поиск в главном окне

В левой панели есть строка быстрого поиска.

### Что она делает

Она не фильтрует дерево полностью, а:

### Почему так сделано

Это лёгкий и быстрый способ навигации без перегрузки модели дерева дополнительной логикой фильтрации.

## 27. Почему проект устроен именно так

### Причина 1. Разделение ответственности

Каждый класс отвечает за свою область:

- `DatabaseManager` — данные и права;
- `TreeModel` — представление дерева;
- `MainWindow` — взаимодействие с пользователем;
- диалоги — локальные задачи.

### Причина 2. Соответствие архитектуре Qt

Qt хорошо работает, когда:

- данные отделены от представления;
- интерфейс построен на моделях;
- формы создаются через `.ui`.

- менять БД-логику отдельно;
- расширять поиск;

## 30. Ограничения текущей реализации

Проект уже функционален, но в нём есть естественные ограничения учебной версии:

- пароль хешируется без сложной адаптивной схемы;
- поиск реализован через `LIKE`, а не через полнотекстовый индекс;
- атрибуты хранятся простым текстом, а не нормализованной схемой;
- одновременная совместная работа для PostgreSQL специально не реализована, так как проект использует SQLite;
- автоматического журналирования изменений нет.
